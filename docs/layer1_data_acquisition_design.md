# データ取得層（Layer1）詳細設計書

作成日: 2026-07-18
対象: AI投資アシスタント Ver3設計書 の一部（Layer1単独の詳細設計）
位置づけ: Ver2設計書で確定した④の未決定事項（評価軸6軸＋総合／選択肢A採用／J-Quants無料継続／重み自動調整なし）を前提に、実装Phase1に着手する前の仕様確定ドキュメント。

---

## 1. 目的・スコープ

Layer1は「日本株・米国株・マクロ・ニュースの生データを、複数のデータソースから優先順位とフォールバックに従って取得し、後段（Layer2分析層）が扱いやすい正規化された形式で提供する」ことのみを責務とする。**指標計算やスコアリングは一切行わない**（それはLayer2の責務）。

本書は以下を確定する。

1. 各データ種別（日本株／米国株／マクロ／ニュース）のデータソース候補と採用理由・代替候補
2. Repositoryパターンによるデータ取得の抽象化設計
3. API障害時のフォールバック設計
4. レート制限対策
5. ローカルキャッシュ・履歴保存方針
6. 将来J-Quants有料プランへ移行する際の変更点
7. Phase1の完了基準

実行基盤はv1/v2設計書の方針通り、GitHub Actions（またはPhase0で決定する同等の外部Python実行環境）を前提とする。このためpipの利用制限は受けず、必要なライブラリ（requests, pandas等）は自由に利用できる。

## 2. データソース候補・採用理由・代替候補

### 2-1. 日本株

| 候補 | 採否 | 理由 |
|---|---|---|
| **J-Quants API（Freeプラン）** | **採用（第一候補）** | JPX公式データ、銘柄マスタ・株価四本値・財務情報・決算発表予定日・取引カレンダーが無料で取得可能。ただし**株価データは12週間遅延**する点が最大の制約。 |
| Web検索（WebFetch/WebSearch経由） | **採用（フォールバックのみ）** | J-Quantsで取得できない場合の代替。ただし個別TSE銘柄の当日株価カバレッジは実証実験で不安定であることを確認済み（Google FinanceのGOOGLEFINANCE関数でも個別東証銘柄はほぼ`#N/A`だった）。 |
| yfinance（GitHub Actions上で技術的には利用可） | **不採用** | Yahoo Finance非公式スクレイピングであり、Yahoo側の仕様変更で頻繁に破損する保守リスクがある。また利用規約上のグレーゾーンでもあり、公式APIであるJ-Quantsを優先するという既定方針と整合しない。 |
| 証券会社API（SBI証券・楽天証券等） | **不採用（将来検討）** | 多くは口座保有者の取引執行を主目的としたAPIで、市場データ配信目的での常時自動アクセスが規約上想定されていない場合が多い。将来、実売買の自動化まで踏み込む場合に再検討。 |

**既知の制約と対応方針（重要）**：J-Quants Freeプランの12週遅延により、**当日時点の日本個別株の価格・出来高・テクニカル指標は、J-Quantsからは実質取得不能**である。この制約下では、
- 日本個別株の「当日」テクニカル分析は、Web検索フォールバックに依存せざるを得ないが、そのカバレッジ・精度は保証できない。
- 取得できた場合でも精度に不確実性が残るため、**日本個別株について当日データの取得元がWeb検索フォールバックになった場合は、当該銘柄をその日の候補から除外し「データ不足」として扱う**データ品質ゲートを設ける（無理に不確実なデータで提案を作らないという既存の運用方針と整合させる）。
- J-Quantsは当面、(a) 銘柄マスタ・取引カレンダー・決算発表予定日の取得、(b) 12週以上前の確定済みデータを使った**バックテスト用の履歴データ蓄積**、の2用途に限定して活用する。

| 候補 | 採否 | 理由 |
|---|---|---|

## 3. Repositoryパターンによる抽象化設計

### 3-1. 全体構成

データ種別ごとに共通の抽象インターフェース（Repository）を定義し、具体的なデータソースへの依存を呼び出し側（Layer2分析層）から完全に隠蔽する。「将来データソースを差し替え可能にする」というご要望を満たすため、**呼び出し側は常にRepositoryインターフェースのみを参照し、どのAPIを使っているかを一切知らない**構造にする。

```
                         ┌───────────────────────┐
                         │   Layer2（分析層）      │
                         │  Repositoryのみを参照    │
                         └───────────┬───────────┘
                                     │ 呼び出し
                                     ▼
                    ┌────────────────────────────────┐
                    │   RepositoryFactory               │
                    │  config/api_sources.yaml を読み、  │
                    │  Fallback+Cacheを組んだ            │
                    │  Repositoryインスタンスを組み立てる  │
                    └───────────────┬────────────────┘
                                    ▼
                 ┌───────────────────────────────────┐
                 │  CachingRepositoryDecorator          │  ← キャッシュ確認（§6）
                 └───────────────────┬───────────────────┘
                                     ▼
                 ┌───────────────────────────────────┐
                 │  FallbackChainRepository              │  ← 障害時の順次フォールバック（§5）
                 │   ┌───────────┐ ┌───────────┐        │
                 │   │ 第一候補   │→│ 第二候補   │→ ...   │
                 │   └───────────┘ └───────────┘        │
                 └───────────────────────────────────┘
```

### 3-2. インターフェース定義（設計仕様であり、実装コードではない）

以下は各Repositoryが備えるべきメソッドのシグネチャ（契約）を示す設計仕様である。実際のクラス実装はPhase1で行う。

**`MarketDataRepository`（日本株・米国株共通の抽象契約）**

- `get_daily_prices(ticker, start_date, end_date) -> PriceSeries`：日次OHLCV（始値・高値・安値・終値・出来高）の時系列を返す。
- `get_fundamentals(ticker) -> FundamentalSnapshot`：PER/PBR/ROE等の計算に必要な原データ（EPS、純資産、当期純利益等の生数値）を返す。比率計算はLayer2で行うため、ここでは加工前の値のみ返す。
- `get_listed_universe() -> List[TickerInfo]`：銘柄マスタ（銘柄コード・銘柄名・業種・上場市場等）。
- `get_trading_calendar() -> List[date]`：取引日カレンダー。
- `get_earnings_calendar() -> List[EarningsEvent]`：決算発表予定日（可能なら実績日も）。
- 戻り値には必ず`DataFetchMeta`（使用ソース名、取得時刻、遅延の有無、成功/失敗ステータス）を付随させる。

**`NewsRepository`**

- `fetch_news(query_or_tickers, since, until) -> List[RawNewsItem]`：記事本文・タイトル・公開日時・情報源URLを返す（要約や重要度付けはLayer3の責務でありここでは行わない）。

**`MacroRepository`**

- `get_series(series_id, start_date, end_date) -> TimeSeries`：FRED系列IDを指定して時系列を返す。

**`FallbackChainRepository`**（上記いずれの抽象型にも被せられる共通コンポーネント）

- 同一インターフェースを実装する複数の具体Repositoryを優先順位付きリストとして保持する。
- 各メソッド呼び出し時、リストの先頭から順に呼び出し、例外の種類に応じて次の候補にフォールバックするか、即座に処理を打ち切るかを判定する（§5参照）。
- どのソースが最終的に使われたかを`DataFetchMeta.source_used`に記録する（Layer4の完全ログ化要件と直結）。

**`CachingRepositoryDecorator`**（同上）

- 任意のRepositoryをラップし、呼び出し前にローカル/リポジトリ内キャッシュを確認、ヒットすればAPI呼び出しをスキップする（§6参照）。

**`RepositoryFactory`**

- `config/api_sources.yaml`を読み込み、データ種別ごとに「どの具体Repositoryを、どの順序で、キャッシュ有無をどう設定してチェーンに組み込むか」を決定し、完成したRepositoryインスタンスを返す。**新しいデータソースを追加する場合、対応する具体Repositoryクラスを1つ実装し、YAMLに1行追記するだけで済み、他のコードは一切変更不要**とする。

### 3-3. `config/api_sources.yaml` の設計イメージ（設定例）

```yaml
japan_equity:
  chain:
    - name: jquants
      plan: free            # free / light / standard / premium
      price_delay_weeks: 12  # freeプラン時のみ有効。有料化時に0へ
      rate_limit_per_minute: 60
    - name: web_search_fallback
      note: "個別TSE銘柄の当日データはカバレッジ不安定。取得成功でもdata_quality_gateで要検証扱い"

us_equity:
  chain:
    - name: alpha_vantage
      rate_limit_per_day: 25
      rate_limit_per_minute: 5
      reserved_for: ["earnings_surprise", "final_candidates_detail"]  # 希少枠を何に優先配分するか
    - name: twelve_data
      rate_limit_per_day: 800
      role: "primary_bulk_scan"  # 広い銘柄スクリーニングの主力

macro:
  chain:
    - name: fred

news:
  chain:
    - name: newsapi
      rate_limit_per_day: 100
      environment: development_only   # 本番の毎朝実行での常用は規約上非推奨（Ver2から継続の注意事項）
    - name: gdelt
      role: "production_primary"
    - name: web_search_fallback
```

この設定に基づき、`RepositoryFactory`が実行時にチェーンを自動構築する。

## 4. データ正規化スキーマ

Repositoryの戻り値は、取得元によらず共通の型に正規化する（Layer2以降がソース差異を意識しないための最重要ポイント）。

- `PriceSeries`：date, open, high, low, close, volume の配列＋ticker＋通貨単位
- `FundamentalSnapshot`：ticker、決算期、EPS、純資産、当期純利益、売上高、営業利益、営業CF、設備投資、有利子負債、総資産、配当金 等の生数値（比率はLayer2で計算）
- `TickerInfo`：ticker、銘柄名、業種コード、上場市場、時価総額
- `RawNewsItem`：title, body, published_at, source_url, source_name
- `TimeSeries`（マクロ用）：series_id, date, value
- 共通メタ：`DataFetchMeta { source_used, fetched_at, is_delayed, delay_reason, success, error_detail }`

## 5. API障害時のフォールバック設計

### 5-1. エラー分類

障害を機械的に区別し、対応を変える。

| エラー種別 | 例 | フォールバック動作 |
|---|---|---|
| 一時的エラー | タイムアウト、5xx、接続エラー | 同一ソースに対して指数バックオフで即時リトライ（§6参照）。上限回数を超えたら次候補へフォールバック |
| レート制限超過 | 429、または残クォータ枯渇 | 即座に次候補へフォールバック（このソースへの追加リトライはしない） |
| 認証・設定エラー | 401/403、APIキー不備 | 次候補へフォールバックしパイプライン自体は継続するが、**run_logに`severity: critical`として記録**し、通常のフォールバックログより目立たせる（キー失効等の見落とし防止） |
| データ不存在 | 該当銘柄が存在しない、404 | フォールバックしない（次候補でも存在しない可能性が高いため）。「対象外銘柄」として記録 |
| 全候補失敗 | チェーン内の全てのRepositoryが失敗 | 当該データ項目は「取得不可」としてsnapshotに明記。Layer5（AI判断層）はこの項目が欠落した銘柄・資産クラスを「様子見」として扱う |

### 5-2. アラート方針

`critical`severityのログが1件でも記録された日は、レポート冒頭（またはGoogle Driveの別ファイル）で明示的に「本日、認証エラー等の異常を検知」と目立つ形で報告する。Layer5のプロンプトにもこの情報を渡し、AIが「データ品質に懸念がある日である」ことを踏まえた慎重な判断をできるようにする。

## 6. レート制限対策

### 6-1. 既知の制限値（実装時に最新値を再確認すること）

| ソース | 制限 |
|---|---|
| J-Quants | 目安1分60リクエスト（コミュニティ情報。プラン間差異は要現地確認） |
| Alpha Vantage（無料） | 1日25件、1分5件 |
| Twelve Data（無料） | 1日800件 |
| FRED | 明確な公開上限なし（過度な帯域利用は禁止） |
| NewsAPI（無料） | 1日100件 |
| GDELT | 明確な公開上限なし（節度あるアクセス頻度を推奨） |

### 6-2. 対策

- **日次クォータトラッカー**：各Repositoryに対し、当日すでに何件呼び出したかをrun内で計測し、上限に近づいたら自動的に次候補へ切り替える、または残クォータが必要な処理（例：Alpha VantageのEARNINGSエンドポイント）専用に予約する（§3-3の`reserved_for`設計）。
- **希少枠の優先配分**：Alpha Vantageの1日25件という制約を踏まえ、**まずTwelve Dataで広くスクリーニングし、最終候補（上位数銘柄）に絞った段階でのみAlpha Vantageを使い、決算サプライズ等のAlpha Vantage固有データを補完する**、という2段階設計を採用する。これによりご指示の優先順位（Alpha Vantage第一候補）を「固有データの取得元」として尊重しつつ、実務上のクォータ制約を回避する。
- **バックオフ＋ジッタ**：一時的エラー時はリトライ間隔を指数的に増やす（例：1秒→2秒→4秒、最大3回）。429（レート制限）の場合はJ-Quantsの実例のように30秒単位の線形バックオフ等、ソースごとの推奨値に合わせる。
- **バッチ/複数銘柄同時取得の活用**：Twelve Data等、複数ティッカーを1回のリクエストでまとめて取得できるエンドポイントがある場合は積極的に利用し、実消費リクエスト数を削減する。
- **スロットル（レートリミッタ）**：全Repository共通のユーティリティとして、設定された`rate_limit_per_minute`を超えないよう呼び出し間隔を制御する仕組みを設ける。

## 7. ローカルキャッシュ・履歴保存方針

### 7-1. キャッシュの二層構造

1. **実行内メモリキャッシュ**：同一実行（1回のGitHub Actions run）内で同じ(ticker, date)への重複リクエストを避ける。テクニカル・ファンダメンタル双方が同じ価格データを参照するケースなどで有効。
2. **永続キャッシュ（履歴データストア）**：日をまたいで再利用し、かつバックテスト用データの蓄積を兼ねる。

### 7-2. 永続化の実装方針（確定）

- 確定済み日次データ（過去の取引日のOHLCV・ニュース要約・マクロ指標）は、一度取得すれば**二度と変化しない**ため、`(source, ticker, date)`をキーに一度取得したら再取得しない設計とする。
- **永続化先はGoogle Driveに一本化する。**GitHubリポジトリにはソースコード・設定ファイル（`config/`配下等）のみを管理し、実行結果・時系列データ・スナップショット・ログ・キャッシュ類は一切コミットしない（リポジトリ肥大化・履歴管理の困難化を避けるため）。
- GitHub Actionsの実行時は、毎回Google Driveから直近のキャッシュ状態（差分取得に必要な「最終取得済み日付」等のインデックス情報）を読み込み、実行終了時にDriveへ書き戻す形で、ステートレスなCI環境上でも永続キャッシュを実現する。
- ファンダメンタルデータ（決算情報等）は日次より変化頻度が低いため、TTL（例：7日）を設けて期限切れ時のみ再取得する。
- ニュースはキャッシュ対象外（常に最新を取得）とするが、記事URL/ハッシュで重複排除は行う。

### 7-3. バックテストとの関係

この永続キャッシュ機構は、Ver2で「バックテスト基盤を前倒し（Phase2）」とした方針と直結する。日々蓄積される確定済み価格・指標データがそのままバックテストの入力データセットになるため、**キャッシュ層の設計＝バックテスト用データレイクの設計**として一体的に扱う。

## 8. 将来J-Quants有料プランへ移行する場合の変更点

Repositoryパターンで抽象化しているため、影響範囲は最小限に設計する。

- `JQuantsRepository`実装クラス自体の変更は**不要**（同一API・同一認証方式のため）。
- 変更が必要なのは`config/api_sources.yaml`の`plan`値と`price_delay_weeks`値のみ（例：`plan: light`, `price_delay_weeks: 0`）。
- §2-1で設けた「Web検索フォールバックに落ちた日本個別株は除外」というデータ品質ゲートは、`price_delay_weeks: 0`になった時点で発動条件（12週遅延データしか無い状態）自体が発生しなくなるため、**ロジック変更なしで自然に無効化**される。
- Standard/Premiumプランで追加解放されるフィールド（信用取引週末残高、財務諸表BS/PL、配当金情報等）は、Ver2で設計した「フィールドごとの利用可否フラグ」を通じてLayer2側が自動的に使い始める設計にしておくため、**Layer2側の追加実装は基本的に不要**（フラグが立てば計算が有効化されるよう最初から実装しておく）。
- 移行時の作業は実質的に「契約変更」「APIキー/トークンの更新」「config 1ファイルの2〜3行変更」のみで完結する設計とする。
